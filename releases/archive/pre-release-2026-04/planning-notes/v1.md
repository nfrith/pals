# ALS Lifecycle Policy v1

This file captures a revised take on ALS lifecycle policy based on the added assumption that ALS may eventually support multiple storage backends.

Examples include:

- markdown files
- JSONL or other structured file stores
- SQLite
- Postgres

This is still a working position, not a settled spec decision.

## Core Revision

The lifecycle policy should remain explicit, staged, and operator-driven.
What changes is the execution model.

ALS upgrade semantics should be backend-neutral.
ALS upgrade execution should be backend-specific.

That means the upgrade system should not be designed as "markdown rewrite tooling" with a few future exceptions.
It should be designed as a canonical ALS upgrade planner with pluggable backend executors.

## Keep The Core Lifecycle

The high-level lifecycle still holds:

1. `check`
2. `dry-run`
3. `prepare`
4. assisted resolution
5. `resume`
6. `apply`

This should remain true whether the system is stored as markdown files or in a database.

## Versioning Still Needs Multiple Axes

The revised model still needs separate clocks.

1. `alsc` binary version
2. `als_version`
3. storage adapter contract version
4. backend engine version or support range

### `alsc` binary version

- Semver is appropriate here.
- Tooling can evolve quickly.

### `als_version`

- `als_version` remains a coarse authored-source or semantic compatibility epoch.
- It should not bump for every additive feature.
- It should bump when authored ALS compatibility changes enough that official upgrade handling is required.

### Storage adapter contract version

- ALS needs a way to version how a backend adapter maps ALS concepts into a concrete storage system.
- This is separate from `als_version`.
- A markdown adapter, a JSONL adapter, and a Postgres adapter may each have their own evolving contract surface.

### Backend engine version

- Some backends bring their own infrastructure lifecycle.
- SQLite library versions and PostgreSQL major versions are examples.
- ALS should be able to express support ranges and prerequisites for these engines.

## Four Distinct Upgrade Layers

ALS should explicitly distinguish:

1. ALS language upgrade
2. storage adapter upgrade
3. backend engine upgrade
4. system or module model migration

These are related, but they are not the same thing.

### ALS language upgrade

- Changes to ALS semantics or source compatibility
- Example: old source construct removed, new official contract replaces it

### Storage adapter upgrade

- Changes to how ALS concepts are persisted or projected into a backend
- Example: markdown serializer changes, SQL schema layout changes, JSONL projection changes

### Backend engine upgrade

- Changes owned by the backend platform itself
- Example: PostgreSQL major upgrade, dropped feature support, changed extension behavior

### System or module model migration

- Domain changes inside one ALS system
- Example: module field rename, enum expansion, record-body reorganization

## Canonical Model First

ALS upgrades should be planned against a canonical ALS representation, not directly against backend-native storage syntax.

Examples:

- markdown adapter parses frontmatter and body into canonical ALS records
- JSONL adapter parses objects into canonical ALS records
- Postgres adapter projects tables and rows into canonical ALS records and relations

The planner reasons over the same logical model in every case.
Only the executor differs.

This is the only path that keeps upgrade semantics universal.

## Adapter Responsibilities

A storage adapter should expose enough capability for upgrades to be real.

Minimum responsibilities:

- discover system contents
- read into canonical ALS representation
- validate backend-specific prerequisites
- create a staged future state
- apply deterministic rewrites
- surface diffs or change reports
- support validation against the staged future state
- apply live cutover
- expose rollback behavior or rollback limits

Useful capability flags:

- `supports_snapshot`
- `supports_atomic_apply`
- `supports_rollback`
- `supports_clone_dry_run`
- `supports_structural_diff`

Different backends will implement these differently.

## Staging Must Be Backend-Specific

Staging remains mandatory, but its implementation depends on the backend.

Examples:

- markdown: git worktree, clone, or staged future directory
- JSONL: copied dataset or staged future directory
- SQLite: copied database file
- Postgres: disposable database, disposable schema, transactionally isolated environment, or a managed clone

The lifecycle should require staging.
The backend adapter decides how staging is achieved.

## Compatibility Windows Still Make Sense

The compatibility-window idea still works in this broader future.

Suggested posture:

1. new ALS construct or adapter contract lands
2. old construct remains temporarily supported
3. validator emits warnings
4. upgrade path is available
5. next compatibility epoch removes the old construct

This remains a compiler and tooling policy, not a commitment to permanent support for old systems.

## Hop-By-Hop Upgrades Still Make Sense

The canonical upgrade path should still be sequential by compatibility epoch.

If a system is on ALS `v1` and the latest is `v3`, the default path should remain:

1. `v1 -> v2`
2. `v2 -> v3`

If a backend adapter also changed materially, that can create additional staged steps.

The CLI may still expose `--to latest`, but internally it should compose official hop-by-hop paths.

## How Backend Prerequisites Should Work

ALS should not pretend every backend engine upgrade is owned by ALS tooling.

Instead, ALS should:

- know what engine versions are supported by a given adapter contract
- detect when the live system is outside that range
- report prerequisites clearly
- block or stage appropriately when those prerequisites are not met

Example:

- ALS system uses a Postgres adapter
- adapter supports PostgreSQL `>=16 <18`
- live system is on PostgreSQL `15`

Then `alsc system upgrade --check` can report:

- ALS language upgrade available
- backend prerequisite unmet
- PostgreSQL engine upgrade required before target adapter path can run

This is better than either:

- silently ignoring backend reality
- pretending ALS fully owns database infrastructure upgrades in the first iteration

## What Users Should See

Users should be able to ask one clear question:

- what is the state of my ALS system lifecycle

That answer should include more than just `als_version`.

Useful reported state includes:

- current `als_version`
- latest supported or recommended `als_version`
- active storage kind
- adapter contract version
- backend engine family and version
- whether an ALS upgrade path exists
- whether backend prerequisites block it
- whether warnings indicate a compatibility window is closing

## Suggested CLI Posture

Normal validation should remain low-noise and non-forcing.

Examples:

- validation passes and reports that a newer ALS version exists
- validation passes and reports that the storage adapter is deprecated
- validation passes and reports that the backend engine is outside the recommended range

Explicit lifecycle commands should provide the full picture:

- `alsc system status`
- `alsc system upgrade --check`
- `alsc system upgrade --dry-run`
- `alsc system upgrade --prepare`
- `alsc system upgrade --apply`

## What Changes Relative To v0

This version changes one important assumption from `v0.md`:

- lifecycle policy cannot be framed only around authored files and file rewrites

The lifecycle model survives.
The implementation target shifts from file codemods to canonical-model planning plus adapter execution.

## Recommended Direction

ALS should move toward this shape:

- `als_version` stays coarse
- compiler semver stays separate
- adapter contract versioning becomes explicit
- backend engine support becomes visible
- compatibility windows remain short
- upgrades remain explicit and operator-driven
- staging remains mandatory
- deterministic and assisted upgrade handling remain first-class

## Open Questions

- where storage backend metadata should live in ALS system configuration
- whether adapter contract version is declared by the system, inferred from the backend plugin, or both
- how much of backend engine support should be modeled in core ALS versus adapter plugins
- whether all backends must support the same rollback guarantees
- whether backend capability flags belong in machine-readable validation output
- how direct skip-path upgrades should work when both ALS epoch and adapter contract changed

## Bottom Line

Supporting alternate storage systems does not invalidate the upgrade model.

It does invalidate a markdown-specific implementation of that model.

The correct long-term shape is:

- one universal ALS lifecycle policy
- one canonical upgrade planner
- many backend-specific adapters and executors
- explicit reporting of ALS epoch, adapter contract, and backend prerequisites
