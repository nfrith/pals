# ALS Lifecycle Policy v0

This file captures one provisional take on ALS lifecycle policy based on discussion during the 2026-03-29 session.

It is not a settled spec decision.
It is a working position to compare against later drafts.

## Core Position

- ALS should not force upgrades on users.
- ALS should let operators choose when to upgrade their systems.
- ALS should not support old source contracts forever.
- ALS should use short compatibility windows instead of permanent backward support.
- ALS system upgrades should be explicit, staged, and auditable.
- ALS should prefer hop-by-hop upgrade paths over large direct jumps.

## Separate The Clocks

ALS currently risks collapsing too many meanings into "version."
These should be separated.

1. `alsc` binary version
2. `als_version` in `.als/system.yaml`
3. upgrade-path catalog versioning

### `alsc` binary version

- Semver is appropriate here.
- The binary can ship frequently.
- New compiler features, diagnostics, and workflow improvements belong here.

### `als_version`

- `als_version` should be a coarse authored-source compatibility epoch.
- It should move slowly.
- It should change only when authored ALS source compatibility changes enough that official upgrade tooling is required.
- Not every ALS feature addition should force an `als_version` bump.

### Upgrade-path catalog

- The compiler should ship a local catalog of supported upgrade paths such as `v1 -> v2`.
- This is how the tool knows what rewrites, warnings, and review steps exist.
- This should not require network access.

## What Counts As A New ALS Version

These should usually stay inside the current ALS version:

- additive language features
- new field types
- new list-item capabilities
- new diagnostics
- stronger tooling or reporting
- new downstream projection behavior that does not invalidate existing authored source

These should usually force a new ALS version:

- removal of old authored syntax
- replacement of one authored source contract with another
- semantic changes that make previously valid authored source invalid
- source rewrites that require official deterministic or assisted upgrade handling

The practical test is:

- can an older valid ALS system remain valid authored source under the new rules without rewrite?

If yes, this is probably not a new `als_version`.
If no, it probably is.

## Compatibility Window Policy

ALS should use compatibility windows at the compiler layer, not permanent language support in the spec.

Suggested shape:

1. new construct lands
2. old construct remains temporarily accepted
3. validator emits warnings and upgrade guidance
4. official upgrade path is available
5. next ALS epoch removes the old construct

This avoids two bad outcomes:

- forcing upgrades at the worst possible moment
- supporting ancient ALS source indefinitely

This is not a mixed-version runtime model.
One system still targets one active `als_version` at a time.

## Upgrade Model

ALS system upgrades should feel closer to governed migrations than to "run one command and hope."

Recommended phases:

1. `alsc system upgrade --check`
2. `alsc system upgrade --dry-run --to <version>`
3. `alsc system upgrade --prepare --to <version>`
4. human or AI resolves assisted items in the staged future state
5. `alsc system upgrade --resume --to <version>`
6. `alsc system upgrade --apply --to <version>`

Important properties:

- live authoritative files are not partially rewritten during discovery
- deterministic rewrites can be staged automatically
- unresolved assisted items produce explicit review artifacts
- `als_version` flips last
- target-version validation happens before live cutover

## Incremental Upgrades

The canonical model should be hop-by-hop upgrades.

If a system is at `als_version: 1` and latest is `3`, the normal path should be:

1. `1 -> 2`
2. `2 -> 3`

The CLI may offer `--to latest`, but the implementation should compose sequential upgrade paths by default.

Reasons:

- each hop has clearer semantics
- each hop has a bounded rewrite surface
- assisted decisions from one hop may affect the next
- auditability stays understandable

Direct skip-path upgrades can be added later if experience justifies them, but they should not be the first model.

## How Users Learn An Upgrade Exists

This should be low-noise and local.

When running validation, the compiler can report:

- current system `als_version`
- latest supported or recommended `als_version`
- whether an official upgrade path exists
- deprecation status for the current system version

Example posture:

- validation still succeeds
- output says an upgrade is available
- user is not forced to upgrade immediately

This is a better fit for ALS than disruptive forced-update behavior.

## Binary Updates vs System Upgrades

These should remain distinct.

Binary updates:

- are about the tool release
- belong to package managers or explicit update workflows
- may move much faster than ALS source epochs

System upgrades:

- are about a particular ALS-authored system
- are driven by explicit operator choice
- use the compiler's local upgrade catalog

The tool should not blur these together.

## Compatibility Metadata

Every significant language change should be classifiable beyond just a version number.

Useful metadata fields include:

- `compatibility_class: additive | conditionally_breaking | breaking`
- `rewrite_required: none | mechanical | assisted`
- `data_impact: none | possible | required`

This metadata should drive:

- validator warnings
- upgrade planning
- machine-readable preflight reporting
- generated review packets for human or AI assistance

## Short-Term Direction

The best near-term shape is:

- keep `als_version` coarse
- keep compiler semver separate
- add compatibility-window support through warnings and local upgrade metadata
- build official hop-by-hop upgrade paths
- stage deterministic rewrites before live cutover
- make assisted review a first-class part of the upgrade lifecycle

## Risks And Open Questions

- how long should one compatibility window last
- how many older `als_version`s should one compiler support at once
- whether warning budgeting should become mandatory before an epoch is retired
- whether upgrade staging should live in a clone, worktree, or dedicated in-repo future-state directory
- whether direct `v1 -> v3` shortcut paths should ever be official

## Bottom Line

ALS should aim for deliberate upgrades, not forced upgrades.

The right model is:

- rare authored-source epochs
- faster-moving compiler releases
- short compatibility windows
- explicit staged upgrades
- hop-by-hop official paths
- no permanent obligation to support old ALS source forever
