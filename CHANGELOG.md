### ALS-093
- Compatibility: refresh_required, additive
- Summary: Scope the ALS-090 primary-clone-convergence pre-commit guard to the authoritative primary worktree only, eliminating the `dispatch_failed_dirty` and AAT `commit-failed` incidents that surfaced on 2026-05-10 (ALS-091 changelog $4.13 + ALS-092 AAT $104.06 + secondary downstream cleanup) when the shared submodule pre-commit hook fired in dispatch-isolation worktrees and no-upstream AAT/update-transaction fixtures whose commit-time questions the guard could not answer. ALS-093 keeps the ALS-090 convergence primitive and the publish follow-through unchanged; the only behavioral narrowing is at the commit site, and the move is delivered as dispatcher canonical `v19 → v20` so every installed copy lands the new hook in lockstep. Five coupled changes ship together. (1) Hook scripting change in `nfrith-repos/als/delamain-dispatcher/src/primary-clone-convergence.ts::buildHookContents()`: the generated `pre-commit` body now resolves `caller_root=$(git rev-parse --show-toplevel 2>/dev/null || pwd -P)` and passes `--invoking-repo-root "$caller_root"` to the helper. The hook still chains user hooks the same way, but it no longer assumes the configured authoritative root is the worktree that actually invoked the commit. (2) CLI scoping change in the same file's `guard` subcommand: the CLI resolves the configured authoritative root via `gitRepoRoot(resolve(repoRoot))`, resolves the invoking root the same way, exits 0 when the two differ, and only runs `convergePrimaryClone()` when the caller is the authoritative primary worktree. Linked dispatch worktrees and no-upstream AAT/update-transaction fixtures fall through cleanly without writing any pending-convergence ledger and without bypass discipline anywhere in agent prompts. (3) Second safety layer for fixture repos that ARE the configured authoritative root but have no canonical upstream — those are treated as "outside the stale-base guard boundary": the guard now exits 0 when `result.status === "blocked" && result.reason === "missing_canonical_upstream"`, instead of failing the commit. This covers the AAT no-upstream fixture path even when the caller-mismatch layer cannot, and is the new failure-modes-table line that operators should know about. (4) Construct refresh: dispatcher canonical bumped `v19 → v20` with sequential no-op migration `nfrith-repos/als/delamain-dispatcher/migrations/v19-to-v20.ts` (persisted runtime-state shape unchanged); the new v20 fingerprint is registered in `nfrith-repos/als/alsc/upgrade-construct/src/known-fingerprints.ts` so a stock v20 install hashes as known-vendor; the five reference-system Delamain dispatcher copies under `nfrith-repos/als/reference-system/.als/constructs/delamain-dispatcher/<delamain>/` (development-pipeline, run-lifecycle, incident-lifecycle, release-lifecycle, postmortem-lifecycle) are refreshed to canonical v20 in lockstep. (5) Rationale locked in the docs the next reader will already encounter: `als-factory/docs/release-model/update-mechanics/primary-clone-convergence.md` gains an "Authoritative Worktree Boundary" section that explicitly names dispatch worktrees and no-upstream fixtures as non-authoritative commit contexts and tightens the existing pre-commit guard bullet to "but only for authoritative primary worktrees"; `nfrith-repos/als/skills/docs/references/delamain-dispatcher.md` mirrors the rule and links back to that rationale doc, satisfying the architecture's "rationale lives in the reference docs that release and dispatcher workflows already read" requirement. The fix is intentionally a commit-site scope change only — `runPrimaryCloneFollowThrough()` and `als-factory-release-rc`'s convergence step are untouched, ALS-090's converge-or-pend behavior is preserved end-to-end on the publish path, and there are no new env vars, marker files, or `--no-verify` clauses anywhere in agent prompts (`grep -rn "no-verify" .als/modules/als-factory/v3/delamains/als-factory-jobs/agents/` returns nothing; same for `.als/modules/als-factory/v3/skills/`).
